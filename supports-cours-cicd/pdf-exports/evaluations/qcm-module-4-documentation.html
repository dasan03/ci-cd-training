Questions



Question 1



Quel est l'avantage principal d'une documentation structurée des tests automatisés ?

A. Améliorer la maintenabilité et faciliter la collaboration entre équipes
B. Réduire le temps d'exécution des tests
C. Augmenter automatiquement la couverture de code
D. Éliminer le besoin de tests manuels

Explication: Une documentation structurée des tests améliore la maintenabilité en facilitant la compréhension du code de test, accélère l'onboarding des nouveaux développeurs, et permet une meilleure collaboration entre les équipes en créant une base de connaissances partagée.

---

Question 2



Allure Report peut uniquement générer des rapports pour les tests JavaScript.

Explication: Faux. Allure Report est un framework multi-langage qui supporte de nombreux langages et frameworks de test : Java (JUnit, TestNG), JavaScript (Jest, Mocha), Python (pytest), C# (NUnit), Ruby (RSpec), et bien d'autres grâce à ses adaptateurs spécialisés.

---

Question 3



Dans Prometheus, quel type de métrique est le plus approprié pour mesurer le temps d'exécution des tests ?

A. Counter
B. Gauge
C. Histogram
D. Summary

Explication: Un Histogram est le type de métrique le plus approprié pour mesurer les temps d'exécution car il permet de calculer des percentiles (50e, 95e, 99e) et de visualiser la distribution des valeurs dans des buckets prédéfinis, ce qui est essentiel pour analyser les performances des tests.

---

Question 5



Quelle approche de monitoring est recommandée pour détecter les problèmes avant qu'ils impactent la production ?

A. Monitoring réactif avec analyse post-mortem
B. Monitoring proactif avec alertes préventives
C. Monitoring manuel quotidien
D. Monitoring uniquement en cas d'incident

Explication: Le monitoring proactif avec alertes préventives permet de détecter les dégradations de performance, l'augmentation des taux d'échec ou autres anomalies avant qu'elles n'impactent les utilisateurs finaux, permettant une intervention rapide et préventive.

---

Question 6



Les rapports de tests doivent être identiques pour tous les publics (développeurs, managers, équipe QA).

Explication: Faux. Les rapports doivent être adaptés à leur audience : les développeurs ont besoin de détails techniques et de stack traces, les managers de métriques de haut niveau et de tendances, et l'équipe QA d'analyses de couverture fonctionnelle et de patterns d'échec.
